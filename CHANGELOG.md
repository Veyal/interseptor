## [Unreleased]

### Fixed
- **`POST /api/authz` always returned `400 "bad json"`.** `identity.Headers` is stored as a single `"Key: Value\n..."` string, but the `set_authz` tool description told callers to send `headers` as an array or object — any non-string shape failed the whole JSON decode with an opaque error, and there was no way to seed an authz identity from a bearer token pulled out of a login response body. Added a custom `identity.UnmarshalJSON` that accepts a string, `["Key: Value", ...]`, or `{"Key":"Value"}` and normalizes to the canonical string form; the 400 now also includes the real decode error. Fixes [#1](https://github.com/Veyal/interceptor/issues/1). (`internal/control/authz.go`.)
- **Map domain selector collapsed to a single host.** The endpoint Map sent the selected domain as a server-side `host=` filter on every `/api/endpoints` load, but the domain dropdown is built from the fetched endpoints — so once a domain was selected, any reload (a background `flow.new`/refresh) re-fetched only that host, rebuilt the dropdown with just that one domain, and left the user unable to switch to any other host. The `host=` filter is now applied **only** during a body/header search (where scoping is intended); a plain load always fetches all hosts, so the dropdown stays complete and domain filtering happens client-side. (`internal/control/ui/js/map.js`.)

### Removed
- **Content-discovery (forced-browse) feature.** The built-in directory/endpoint brute-forcer has been removed: its probes polluted Proxy History, and AI agents leaned on it instead of running purpose-built tools. Deleted the `internal/discovery` engine, the Discover UI tab/panel, the `/api/discovery/*` routes, and the MCP tools `start_discovery` / `stop_discovery` / `discovery_state` / `get_discovery_wordlist` / `suggest_discovery_paths`. The MCP RECON guidance now tells agents to run a real forced-browser (feroxbuster / gobuster / ffuf) **through the proxy** so hits still land in History, then triage with `list_flows` / `host_stats`; the autopilot planner no longer proposes `start_discovery`. Existing flows captured by the old feature are untouched (the `FlagDiscovery` bit and its "DSC" history badge are retained so legacy data still labels correctly).

### Added
- **Multiple saved AI providers with one-click switching.** Keep several provider configurations (e.g. Anthropic + an OpenAI-compatible endpoint + GLM) as named **profiles** and switch the active one from Settings → AI assist → *Saved providers*. Activating a profile mirrors its `{provider, key, model, endpoint}` into the canonical `ai.*` settings that everything already reads (assist, autopilot, checks/notes AI), so no consumer needs to change. "Save as provider profile" snapshots the current config (the live key is captured server-side even when the masked key field is left blank). Keys are stored but never returned to the client (`hasKey` is surfaced instead). Endpoints: `GET/POST /api/ai/providers`, `DELETE /api/ai/providers/{id}`, `POST /api/ai/providers/{id}/activate`. (`internal/control/aiproviders.go`.)
- **Collaboration & remote access — share a live session, work a target together.** Interceptor can now be reached securely from outside your machine (no public IP required) so an AI agent on a VPS can drive it, a teammate can watch your live session, and two operators on the same target can merge their work like a lightweight git. All from the UI (Settings → API & MCP → **Share**). Built on the existing loopback trust model, which is preserved byte-for-byte for local use.
  - **Scoped, expiring access keys.** API keys now carry a **scope** (`read` — watch only; `full` — act) and an optional **expiry**. Existing keys migrate to full/never-expire. `store.VerifyAPIKeyScope` enforces both. Create them with a scope + expiry picker in the Keys tab.
  - **Key-authorized remote surface.** The security guard gained a second trust mode: a request carrying a valid key is authorized regardless of Host/Origin (the remote-access path), while an unauthenticated non-loopback request is closed (401, or redirected to a login page for a browser). A read-only key may only read (GET/HEAD + SSE); a full key may mutate. Loopback-with-no-key is unchanged. DNS-rebinding and CSRF stay defeated: the cookie (browser) path additionally requires an `X-Interceptor-CSRF` header, `SameSite=Strict`, and a same-origin Origin on mutations; the bearer path is immune (no ambient credential). A `?token=` query is honored only for the SSE stream (EventSource can't set headers). The `/mcp` tool surface requires a **full** key. The MCP backing bus was moved onto the loopback control address so remote MCP calls don't loop back out over the tunnel and 401 themselves.
  - **Browser login.** A minimal `/login` page mints an httpOnly session cookie (`POST /api/session/auth`, rate-limited per IP; `Secure` set when the request is HTTPS via the tunnel), so a teammate opens your tunnel URL, enters a key, and watches the live session with zero client config. `POST /api/session/logout` clears it.
  - **One-click Cloudflare tunnel** (`internal/tunnel`). Interceptor launches `cloudflared` as a managed child process (a free quick tunnel — no account, no public IP), parses the assigned `https://*.trycloudflare.com` URL from its output, and surfaces it live in the Share panel over a new `tunnel.update` SSE. `cloudflared` is detected on PATH, never downloaded; when absent the panel shows install guidance. Starting a tunnel is **refused unless at least one key exists**, so the surface is never exposed unauthenticated, and it is torn down on shutdown. Endpoints: `GET /api/share/status`, `POST /api/share/start`, `POST /api/share/stop`.
  - **Additive project merge (pull/push).** Two people on the same target can union their captured flows and findings without overwriting anything (`store.MergeFrom`). Content-addressed bodies dedupe automatically; flows dedupe by a content signature (so re-pulling is idempotent); findings are appended with their PoC flow references remapped to the new local ids and deduped by a title/target signature; imported flows are tagged `peer/<label>` for provenance. `POST /api/merge/file` is the push receiver; `POST /api/merge/pull` downloads a peer's full-project export over their tunnel (URL + key) and merges it; `POST /api/merge/push` uploads yours to their receiver. Pull/push run server-side so the peer key never touches the browser. Reuses the existing zip-slip-guarded, member-allowlisted full-project archive machinery.

### Fixed
- **Autopilot produced empty plans on providers that don't emit tool calls.** The planning phase relied entirely on the model calling read-only recon tools (`list_flows`, `host_stats`, …) to discover captured history. Many OpenAI-compatible endpoints (and some models — e.g. Grok via a reseller proxy) never emit `tool_calls`; they return prose instead, even with `tool_choice:"required"`. The planning agent therefore called nothing, hallucinated "no in-scope history available", and returned a **zero-step plan**, so every run finished immediately having done nothing. The plan phase now builds a **deterministic recon digest** (scope rules + per-host traffic + a bounded endpoint sample) directly over the in-process tool bus and injects it into the planning prompt, so planning no longer depends on the model choosing to call tools — tool-capable models still get the tools to dig deeper. A zero-step plan now also emits a loud Activity diagnostic instead of silently finishing. Verified end-to-end against a live project: the model now produces a populated multi-step plan with zero tool calls. (`internal/autopwn/recon.go`, `planPhase`.)
- **Agent-mode replies were capped at the terse prose limit (768 tokens).** The tool-calling planning/verification turns and the final synthesis reused the assistant's short-answer `max_tokens`, which could truncate a multi-step JSON plan or verdict mid-object and fail the caller's parse. Agent turns now use a dedicated, larger ceiling (`agentMaxTokens`). (`internal/aiassist`.)

## [0.28.0] - 2026-07-05

### Added
- **Autonomous AI pentester ("Autopilot") design.** Added [docs/AUTONOMOUS-PENTEST.md](docs/AUTONOMOUS-PENTEST.md), the architecture + phased build plan for an AI agent that reads captured history, autonomously runs active security testing using Interceptor's own tools (in-process access to all MCP tools), tracks every step in the Activity feed with every request visible in History, and files **only machine-verified true-positive findings** via a 4-gate verifier (differential reproduction → adversarial verifier agent → out-of-band proof for blind classes → human confirm for Critical/High). Grounded in a two-pass read-only audit of the existing AI-agent, MCP, active-scan, findings, OOB, and sender machinery. Implementation lands over Phases 0-4 on `feat/autonomous-pentest`.
- **Autopilot Phase 0 — backend seams, schema & verification primitives** (no autonomy wired yet; all additive, TDD):
  - **In-process tool bus:** `mcp.Server.Call(name, args)` invokes any of the ~84 MCP tools in-process by name. `callTool` was refactored to share a private `runTool` helper so in-process calls report to the Activity feed and carry `X-Interceptor-Source: ai` (FlagAI History tagging) identically to JSON-RPC dispatch — the seam the autonomous engine drives all tools through.
  - **Schema:** new per-project SQLite tables `pentest_run` (run lifecycle: status/scope/plan/budget/summary JSON) and `finding_verification` (the machine proof-record behind a `verified` finding — vuln class, which gates passed, repro count, OOB token, PoC flow ids, derived confidence; UNIQUE on `finding_id` for a 1:1 upsert). Both use `CREATE TABLE IF NOT EXISTS` in the always-run `schema`, so existing projects gain them on next open.
  - **OOB token→probe correlation** (`internal/oob`): `MintCorrelated`/`CorrelationFor`/`InteractionsForToken` link a minted OOB token to the run/candidate/probe-flow that injected it — the previously-missing wire that lets blind-vuln probes be proven by a callback to a globally-unique injected URL. Additive, on a separate mutex; the existing interaction ring and manual-OOB flow are unchanged.
  - **Deterministic verification primitives** (`internal/verify`, new package): Gate 1 differential reproduction (`ReproduceDifferential` + pure, table-tested class oracles for reflected-marker / error-signature / boolean-length / timing, requiring the oracle to hold N consecutive times to reject flukes) and the Gate 3 OOB-confirmation mechanism (`ConfirmOOB`). Decoupled via injected `Sender`/`OOBPoller`/clock interfaces — no live network, no real sleeps — so it's unit-testable now and wired to the real sender/OOB in a later phase.
- **Autopilot Phase 1 — provider-agnostic agent loop & the full 4-gate verifier** (still no autonomous engine wired; additive + decoupled, TDD):
  - **`internal/aiagent`** (new): a `ToolCaller` interface with a budgeted tool-calling `Run` loop (hard stops on max steps / tool calls / tokens / wall-clock via an injected `Clock`; `StoppedBy` = done|budget|ctx|error) driving an injected `ToolExecutor` — fully unit-tested against a fake provider + fake executor with no live API.
  - **Multi-provider tool calling.** `internal/aiassist`'s agent path was Anthropic-only; it now admits **GLM** (same Anthropic wire format — unlocked via an allow-list plus the correct Bearer auth) and **OpenAI/OpenRouter** (a real chat-completions `tools`/`tool_calls` agent path, normalized to the same turn shape). The existing in-app "Ask AI (agent mode)" is unchanged for Anthropic and now also works on GLM/OpenAI/OpenRouter.
  - **The 4-gate verifier** (`internal/verify`): top-level `Verify` composes Gate 1 (differential reproduction) → Gate 2 (adversarial verifier agent) → Gate 3 (OOB proof, only for blind classes) → Gate 4 (human confirm, only for Critical/High), short-circuiting on the first failure so unproven candidates never become findings. Emits a `Proof` record (which gates passed, repro count, OOB token, PoC flow ids, a weighted 0-100 confidence) that maps straight onto the `finding_verification` row. Gates 2 and 4 are injected interfaces (concrete impls wired to the agent loop + human-input gate in Phase 2); 14 tests cover every pass/reject/skip path.
- **Autopilot Phase 2 (engine core) — the autonomous orchestration engine** (`internal/autopwn`, new): a scope-gated run lifecycle (planning → execution → 4-gate verification → filing) driven in a goroutine under one cancellable ctx (the kill switch), plus the concrete adapters that couple the Phase-0/1 injected interfaces to real infrastructure (sender, OOB catcher, the MCP in-process tool bus, and the human-input confirm gate). `Start` refuses to run without scope rules; the plan phase uses the budgeted agent loop over read-only recon tools to build an attack plan from captured history; the execution phase drives active tools (active_scan/intruder/authz/discovery) over the tool bus behind a per-host circuit breaker; the verify phase runs each candidate through the 4 gates and files **only** proven findings (`Source:"ai"`, `Status:"verified"`, PoC flows attached, a `finding_verification` proof-record written). Every phase, tool call, candidate, and verdict is recorded to the Activity feed with an intent and broadcast as `autopwn.update`. Request/token/wall-clock budgets and `Stop()` are hard stops. **Safety rails** (hardened after adversarial review): the verifier's re-send path re-checks every probe URL — candidate target, all Gate-1 request variants, and the resolved OOB probe — against the run's *scope snapshot* + an own-listener predicate before sending, so autonomous attack traffic can never leave the authorized boundary; finalization is panic-safe (a panic in any phase ends the run cleanly instead of bricking the engine); and a blind candidate with no configured OOB callback URL is skipped with a loud diagnostic rather than silently failing. End-to-end unit-tested with a fake LLM provider (no live API): proven candidates file, gate rejections and boundary/own-listener/config skips file nothing, `Stop()`/budget exhaustion halt cleanly.
- **Autopilot Phase 2 (control + MCP wiring) — the engine is now reachable.** The Hub lazily constructs the engine (deferred until the control listener is bound, since the in-process tool bus and OOB base both need the server's own address) and exposes it: `POST /api/autopwn/start` (body `{budget:{maxRequests,maxTokens,maxWallMs}, targetHint}` → `{runId, state}`; `409` if a run is active, `400` if no scope rules), `POST /api/autopwn/stop`, `GET /api/autopwn/state`, `GET /api/autopwn/runs`, live progress over the `autopwn.update` SSE event, and MCP tools `autopwn_start`/`autopwn_state`/`autopwn_stop`. The run is started on a background context so it outlives the HTTP request (the kill switch is `Stop()`); the tool bus reaches the ~84 tools over loopback REST (no API key needed — only `/mcp` is key-gated, and the engine calls tools in-process); the tool-caller resolves the currently-configured AI provider fresh per run; human-confirm wraps the existing human-input gate with a 30-minute window; and the OOB base prefers the persisted `oob.baseUrl` (target-reachable) over the loopback fallback so blind classes are verifiable.
- **Autopilot Phase 4 — end-to-end verification.** A real over-the-wire integration test (`internal/autopwn/e2e_test.go`) stands up a deliberately-vulnerable `httptest` server (reflected-XSS, error-SQLi, a non-vulnerable control, and a genuine server-side-fetch blind-SSRF endpoint) and drives the engine through the **real** sender → real differential oracle → real OOB catcher → real finding filing, with only the LLM mocked. It proves: reflected + error candidates each file exactly one machine-verified finding with a proof-record and attached PoC flows (endpoint hit counts confirm the re-sends actually happened over the wire); the non-vulnerable endpoint is rejected at Gate 1 and files nothing; and a blind SSRF is proven only when a real callback reaches the correlated OOB token (confidence 100), while a no-callback variant is rejected. The wired binary was also runtime-smoke-tested end to end on an isolated throwaway project: the scope gate refuses a run with no scope (HTTP 400), and a run with scope but no AI key transitions cleanly to `error` in the plan phase having sent zero requests (no attack traffic before the execute phase).
- **Autopilot Phase 3 — the UI panel.** A new **Autopilot** tab (its own group at the top of the nav rail) built on the redesign primitives: run controls (target hint + request/token/wall-clock budgets, a `.btn-primary` Start gated behind an explicit danger-styled confirm since it launches real autonomous attack traffic, and a `.btn-danger` kill switch), live budget meters (amber ≥80%, red ≥100%), and a live pipeline driven by the `autopwn.update` SSE — phase progression, plan summary, and a per-candidate card showing verifying → ✓ filed (with a deep-link to the finding) / ✗ rejected @ gate / skipped: reason, plus seen/verified/filed/rejected counters and a run-history list. Lazy-loaded on first visit with a nav badge for off-screen updates, mirroring the Discover/Map pattern. The panel restates the glass box: every request is in Proxy History, every step in Activity, and only machine-verified findings land in Findings. The `400` no-scope and `409` run-active API responses are surfaced as clear messages.
- **Ask AI on findings.** An "Ask AI" button has been added to the Findings view. When clicked, the AI is pre-loaded with the finding's context (title, severity, description blocks, impact) and any linked PoC flows, allowing seamless AI analysis of vulnerabilities.
- **Settings search.** A search box atop the Settings nav filters sections by label or body text, so options are discoverable without knowing which group they live in (Esc clears).
- **Danger zone.** Destructive/irreversible settings (Data & Retention deletes, purge, GC) now render in a red-bordered card, visually separated from ordinary configuration.
- **Mobile devices settings tab.** Android (ADB), iOS (Simulator & Device), and iOS jailbroken (SSH) moved out of TLS / CA into their own **Mobile devices** section, so TLS / CA now holds only the CA download, SSL-pinning diagnosis, and trust guide.
- **Command palette coverage.** Added Ctrl+K entries for Mobile devices settings, Compare selected flows, Copy selected flow as cURL, Switch/create project, Run setup wizard, and Toggle theme — closing gaps where screens weren't reachable from the palette.
- **UI redesign roadmap.** Added [docs/UI-REDESIGN-ROADMAP.md](docs/UI-REDESIGN-ROADMAP.md), a three-agent audit (feature inventory, UX/design-system blueprint, performance architecture) synthesized into a phased plan for the control-UI overhaul on `redesign/ui-overhaul`.
