CAPTURE POLICY

On long engagements, save database space by storing only in-scope traffic. Out-of-scope requests are still forwarded — they're just not written to history. Define your target under Target scope; with no scope set, everything is in scope (so this has no effect).

Install & trust this CA on the client to intercept HTTPS. Per-host leaf certificates are minted on demand.

TARGET SCOPE

With include rules, only matching traffic is in scope; exclude always wins; no rules = everything in scope. Scope focuses the ◎ in scope history toggle, the intercept gate, and the scanner. Tip: right-click a flow → Add to scope. Host supports a leading wildcard, e.g. *.acme.com; path is a prefix.

AI ASSIST · bring your own key

Optional. With your own provider key, the inspector's ✨ can explain a request, suggest payloads, or summarize findings. Off until a key is set; the exchange is sent to your chosen provider only when you click ✨.

SESSION · auth headers

Headers auto-applied to every Repeater and Intruder send (which bypass the proxy), so a token stays fresh without re-pasting. One Key: Value per line; these replace any existing value on the outgoing request. Typically an Authorization bearer token or a Cookie.

PROJECT

Each project keeps its own captured history, rules, target scope & custom checks under ~/.interceptor/projects/<name>/. The CA is shared across all projects (~/.interceptor/ca), so you only trust it once. Switching restarts Interceptor on the chosen project.
